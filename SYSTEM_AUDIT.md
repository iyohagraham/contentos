## 1. Executive Summary

### What the platform does
ContentOS is a **single-dashboard "content engine" for faceless short-form video channels.** From one React SPA an operator can:
- Generate a channel **monetization strategy** (brand, content pillars, posting schedule, growth roadmap, product funnel) with AI.
- Generate **video scripts** and **content ideas** with AI.
- Turn a script into a **HyperFrames HTML video composition** (text-on-screen timeline) and preview/download it.
- **Publish / schedule** posts across TikTok / Instagram / YouTube / Facebook (via Postiz).
- Track **analytics** (views, followers, engagement) and **digital-product sales** (guides, templates, courses).
- Manage multiple **channels** and a content **calendar**.

### Target users
Solo creators and small teams running **faceless / automated short-form channels** who want an all-in-one cockpit instead of stitching together a scriptwriter, a designer, a scheduler, and an analytics tool. (Seed data is themed around a "Solo Setup Guide / freelance business" niche.)

### Core business objective
Help a creator **scale multi-platform content production and monetize it** — generate content cheaply with AI, distribute it everywhere, and convert audience into digital-product revenue. The app itself is positioned as the orchestration layer; the money is meant to come from the creator's product funnel.

### One-line status
**Functional and deployed**, running in **demo/localStorage mode** with AI script/strategy generation live (Kimi). Cloud persistence, auth, real social posting, and real AI media generation are **built but not activated** (each blocked on an external account/key). It is a strong *generation + dashboard* product with **no content *research/intelligence* capability yet**.

## 2. Current Architecture

| Layer | Technology | Notes |
|---|---|---|
| **Frontend** | React 18, Vite 5, Tailwind CSS 3 | Single SPA. One ~2.2k-line `src/App.jsx` holds all 9 views. `lucide-react` icons. `date-fns` + `recharts` are dependencies (**recharts is installed but unused** — charts are hand-rolled divs). |
| **Backend** | Vercel Serverless Functions (`api/**/*.js`, Node) | Plain handler functions. `server.js` (Express) mirrors the same handlers for local dev so there's one codebase. Max function duration 30s (`vercel.json`). |
| **Database** | Supabase (PostgreSQL) **or** `localStorage` | Adapter pattern in `src/lib/db/store.js` auto-selects: Supabase when `VITE_SUPABASE_*` are set, else `localStorage`. Currently **localStorage** in prod. |
| **APIs** | REST-ish JSON over `/api/*` | AI generation, analytics, Postiz proxy, social, cron. See §8. |
| **Authentication** | Supabase Auth (email/password) | **Scaffold deployed but dormant** (`src/lib/auth.js`, `src/AuthGate.jsx`). Activates only when Supabase is configured; otherwise no login wall. RLS policies exist in schema. |
| **Hosting** | Vercel | Git-connected repo, but deploys are done via `vercel --prod` CLI (git push does **not** auto-deploy in practice). 1 cron configured. |
| **Storage (assets)** | **None** | No Vercel Blob / S3 / object store. Generated images/video/audio are **external URLs** returned by fal.ai/Postiz; these are ephemeral and not persisted. This is a real gap (see §10). |
| **AI** | Kimi k2.7 (text) + fal.ai (media) | See §6. No embeddings, vector store, or RAG. |

**Rendering reality:** "video" today = an **HTML composition** (GSAP timeline of text clips) generated by `openmontage-bridge.js`. There is **no server-side render to MP4**. The user previews/downloads HTML; nothing produces an actual video file in-app.

## 4. Database Audit

> The schema (`supabase/schema.sql`) defines **8 tables**. **Important:** in the current
> localStorage mode the store (`store.js`) only implements repositories for **6** of them —
> `workspaces, strategies, channels, videos, products, sales`. **`video_posts` and
> `analytics_snapshots` are schema-only: no app code reads/writes them yet.** RLS is enabled
> on all tables; per-table policies now exist for all (the `video_posts`/`sales` deny-all gap
> was fixed this session).

### `workspaces`
- **Purpose:** Top-level brand container; a user can own several.
- **Fields:** `id`, `user_id → auth.users`, `name`, `niche`, `audience`, `brand_color`, `font_family`, `logo_url`, timestamps.
- **Relationships:** 1→N to strategies, channels, videos, products, analytics_snapshots.
- **Usage:** Seeded with one workspace ("Solo Setup Guide"). **Caveat:** the publish flow hardcodes `workspace_id: 'default'` when creating videos, which does **not** match the seeded workspace id → minor multi-workspace inconsistency.

### `strategies`
- **Purpose:** AI-generated channel strategy.
- **Fields:** `id`, `workspace_id`, `brand_name`, `handle`, `tagline`, `positioning`, `content_pillars` (JSONB), `posting_schedule` (JSONB), `growth_roadmap` (JSONB), `product_strategy` (JSONB), `viral_hooks` (JSONB), timestamps.
- **Relationships:** N→1 workspace.
- **Usage:** Written by StrategyView on generate; read-back is light (UI mostly renders the just-generated object).

### `channels`
- **Purpose:** A connected social account.
- **Fields:** `id`, `workspace_id`, `platform`, `handle`, `display_name`, `avatar_url`, `followers`, `status`, OAuth fields (`access_token`, `refresh_token`, `token_expires_at`, `platform_account_id`), `auto_post`, timestamps.
- **Relationships:** N→1 workspace; 1→N video_posts, analytics_snapshots.
- **Usage:** Seeded with 3 channels; drives Dashboard/Analytics/Channels views. **OAuth token columns are never populated** (no connect flow writes them).

### `videos`
- **Purpose:** A video project through its lifecycle.
- **Fields:** `id`, `workspace_id`, `title`, `topic`, `script` (JSONB), `format`, `content_style`, `status` (draft/rendering/ready/scheduled/published/failed), asset URLs (`video_url`, `thumbnail_url`, `audio_url`, `composition_path`), `scheduled_time`, `published_at`, `target_platforms` (JSONB array), timestamps.
- **Relationships:** N→1 workspace; 1→N video_posts; referenced by sales.
- **Usage:** Core entity. Seeded (4) + created by the publish/schedule flow. Calendar buckets by `scheduled_time`/`published_at`. **`video_url`/`thumbnail_url` are essentially always null** (no render/storage pipeline).

### `video_posts` *(schema-only)*
- **Purpose:** Per-platform publish record + per-post analytics (views/likes/comments/shares/saves/clicks/engagement_rate).
- **Relationships:** N→1 video, N→1 channel.
- **Usage:** **Unused.** No store repo, no UI. This is where real per-platform results *should* land once posting+analytics are wired.

### `products`
- **Purpose:** A digital product to sell.
- **Fields:** `id`, `workspace_id`, `name`, `description`, `price`, `currency`, `product_url`, `product_type`, `platform` (gumroad/beacons/stripe/...), `total_sales`, `total_revenue`, `conversion_rate`, timestamps.
- **Usage:** Seeded (3); drives Dashboard + Monetize views.

### `sales`
- **Purpose:** Individual sale, optionally attributed to a video.
- **Fields:** `id`, `product_id`, `video_id` (nullable), `amount`, `currency`, `source_platform`, `customer_email`, `created_at`.
- **Usage:** Store repo exists (`list`/`create`) but **no UI writes sales**; revenue numbers come from `products.total_revenue` aggregates, not from `sales` rows.

### `analytics_snapshots` *(schema-only)*
- **Purpose:** Daily per-channel aggregate (followers/views/likes/clicks/revenue) for trend charts. `UNIQUE(channel_id, snapshot_date)`.
- **Usage:** **Unused.** No code writes snapshots. This is the table a real "growth over time" chart would need; today trend charts are faked.

**DB-level debt:** real numeric history (snapshots, per-post stats) is absent → all "over time" / "+12% change" UI is **fabricated placeholder data**, not derived.

## 6. AI Systems Audit

### Text generation (primary, live)
- **Provider:** Moonshot AI ("Kimi"). Base URL `https://api.moonshot.ai/v1`, accessed through the **OpenAI SDK** (`openai` npm) for API compatibility.
- **Model:** `kimi-k2.7-code-highspeed` (chosen for ~2s latency vs ~26s standard). `temperature: 1` is **required** by this model.
- **Key:** `KIMI_API_KEY` (fallback alias `MOONSHOT_API_KEY`). Validated by `hasValidKey()` (length + not-placeholder).
- **Shared client:** `api/_kimi.js` — `kimiChat(messages, {maxTokens})` + `parseJSON()` which strips ``` fences, extracts the first `{...}` block, and **repairs malformed JSON** with `jsonrepair` (robust against LLM output drift).
- **Prompt systems (3 task-specific, JSON-only):**
  - **Script** — system: "expert viral short-form video scriptwriter, output ONLY JSON". User prompt injects topic/style/niche/audience/length; expects `{hook, body[], cta, estimatedDuration, suggestedVisuals[]}`. `fullScript` is assembled server-side.
  - **Strategy** — system: "expert social media monetization strategist". Expects a fixed `{brand, pillars[4], schedule{4 platforms}, roadmap[3], product{funnel}}` shape.
  - **Ideas** — system: "viral content strategist". Expects `{ideas:[{title,type,description,visual}]}`.
- **Resilience pattern:** every endpoint has a hardcoded **demo object** and returns it (with `demo:true`) when the key is missing or the call throws — the UI never hard-fails on AI.

### Media generation (built, not securely wired)
- **Provider:** fal.ai via `@fal-ai/client`. Key env: `FAL_AI_API_KEY`.
- **Models referenced in `src/lib/fal.js`:** `fal-ai/flux/dev/image` (visuals), `fal-ai/wan` (image→motion; code/comments say **Wan 2.1**), `fal-ai/kokoro` (TTS), `fal-ai/background-removal`.
- **Caveat:** `fal.js` is **frontend** code with `configureFal(key)`; there is **no `/api` route** for media generation. Securely using fal requires a server endpoint (so the key isn't shipped to the browser). Today this path is effectively demo/unused.

### Composition "AI" (deterministic, not a model)
- `openmontage-bridge.js` is **rule-based**, not ML: it maps `{hook, body[], cta}` → a fixed timeline of text clips (intro 2s, hook 3s, each body 4s, cta 3s) and emits GSAP-timeline HTML. No model call.

### Not present (relevant to roadmap)
- ❌ Embeddings / vector store / semantic search
- ❌ RAG (no retrieval corpus at all)
- ❌ Agents / tool-use / multi-step orchestration
- ❌ Transcription (no Whisper or equivalent)
- ❌ Any analysis-of-existing-content model usage (the AI only *creates*, never *studies*)

## 7. Workflow Audit

### A. User flows (synchronous, in-app)
1. **Strategy** → StrategyView form → `POST /api/generate-strategy` (Kimi) → render + persist to `strategies`.
2. **Create video pipeline** (CreateView, 5 steps: Script → Voice → Visuals → Design → Publish):
   - Step 1 Script → `POST /api/generate-script` (Kimi).
   - Steps 2–3 Voice/Visuals → intended fal.ai; currently demo placeholders (no secure route).
   - "Generate composition" → `POST /api/generate-composition` → HyperFrames HTML → iframe (blob URL) preview + download.
   - Step 5 Publish → Postiz (see C).
3. **Content management** → list/search/filter/delete videos.
4. **Calendar** → view by month; drag-drop reschedule (persists raw ISO date); bulk schedule/publish.
5. **Analytics** → `GET /api/analytics?period=` → Postiz aggregate if configured, else mock; platform breakdown from channels.
6. **Channels** → CRUD + auto-post toggle + "sync from Postiz".

### B. Background jobs / scheduled tasks
- **Vercel Cron `/api/cron/process-scheduled`** — configured in `vercel.json` as `0 9 * * *` (**daily 09:00**, *not* the "every 5 min" the code comment claims). Reads `videos` where `status='scheduled' AND scheduled_time<=now`, calls `socialMediaManager.postToMultiple(...)`, updates status. **Requires Supabase** (returns `processed:0` otherwise). **⚠️ Critical flaw:** in a fresh serverless invocation the `socialMediaManager` has **no connections** (nothing ever calls `.connect()`), so posting fails → this job cannot actually publish. It also bypasses Postiz entirely.
- **`scheduler.js`** — an in-memory `setInterval` (60s) auto-posting scheduler that **auto-starts on import**. Only the Express dev server imports it; it's ephemeral and **serverless-incompatible**. Dead in production.

### C. Automations / integrations
- **Publish (real path):** UI → `postizClient` → `/api/postiz/post` → `PostizClient.createPost()` → Postiz public API (`{POSTIZ_URL}/public/v1/posts`, auth header = raw API key). Supports immediate (`type:'now'`) and scheduled (`type:'schedule'`). **This is the wired, working distribution path** (when Postiz is deployed).
- **Direct connectors (`manager` + tiktok/instagram/youtube/facebook):** complete REST clients for TikTok Business / IG Graph / YouTube Data / FB Graph, exposed via `/api/social/*`. **No OAuth onboarding flow exists**, so they're unreachable in practice. This duplicates Postiz and is the source of the cron flaw.

### D. AI workflows
- Topic → script (Kimi) → composition (rule-based HTML). That's the full "AI content" chain today. There is **no** research→analyze→generate loop.

**Net:** real automation today = "generate text with Kimi" + "push to Postiz". Everything else (rendering, media gen, cron publishing, snapshots) is stubbed, dead, or fabricated.

## 10. Missing Components (bottlenecks, debt, scalability)

### Technical debt (fix-soon)
1. **Two competing social systems.** Postiz (wired) vs. direct connectors (dormant). Pick one. Postiz should win; the cron must be rewritten to publish via Postiz, and the direct connectors + `scheduler.js` + `/api/social/*` should be removed or quarantined.
2. **Cron cannot publish.** It uses the unconnected manager and runs daily. Rewrite to query Supabase and call the Postiz client; raise frequency to ~5 min.
3. **No secure media-gen route.** `fal.js` must move behind `/api/generate-visual` / `/api/generate-voice` so the fal key stays server-side. Until then visuals are fake.
4. **Broken generated HTML.** `openmontage-bridge.js` emits `https://cdn.gsap.in Equation/hyperframes-core.js` (corrupted) → compositions don't animate. Fix the CDN/runtime reference (or render server-side).
5. **No object storage.** Generated media + final renders have nowhere to live; fal URLs expire. Add Vercel Blob (now supports public+private) or S3.
6. **No real render-to-MP4.** "Video" never becomes a file. Either integrate HyperFrames/Remotion server render or a render service.
7. **Fabricated metrics in UI.** Hardcoded "+12%/+23%/+34%" deltas and the fallback bar chart present fake growth on a live analytics product — an integrity issue. Drive from `analytics_snapshots` or hide.
8. **Schema/usage drift.** `video_posts` and `analytics_snapshots` unused; `workspace_id:'default'` hardcoded in publish vs seeded workspace id.
9. **`recharts` installed but unused**; health-endpoint field name differs local vs prod (`kimiconfigured` vs `configured`).
10. **No tests / no CI / no lint.** Zero automated safety net on a single 2.2k-line component.

### Scalability concerns
- **`App.jsx` monolith** (~2.2k lines, all views + state) — hard to extend, will become the main velocity bottleneck for new features. Split into routed view modules + a state layer.
- **No auth/multi-tenancy live** — localStorage is single-device; cloud+RLS exist but unactivated. Multi-user can't scale until Supabase is on.
- **Serverless cold starts + 30s cap** are fine for text gen but **insufficient for media gen / rendering** (those need a queue + longer compute, e.g. Vercel Queues + a worker, or an external render service).
- **No caching / rate-limit handling** around Kimi/fal; no idempotency on publish.
- **No observability** (no logging/metrics/error tracking beyond `console.error`).

### Missing systems (net-new)

## 12. Content Intelligence Readiness

**Current readiness: ~0%.** The platform can *generate* content from a topic string but has **no ability to ingest, store, search, or learn from existing content.** Every capability below must be added from scratch.

| Capability | Supported now? | What's missing / needs adding |
|---|---|---|
| **YouTube research** | ❌ | YouTube Data API client (search, channel/video stats), an ingestion job, and a `research_sources` store. None exist. |
| **Instagram research** | ❌ | IG Graph (own assets only) is insufficient for competitor research; needs a 3rd-party data/scrape provider (e.g. Apify) + ingestion. |
| **TikTok research** | ❌ | TikTok research API / 3rd-party scrape provider + ingestion. The existing TikTok connector is *publish-only* (Business API), not research. |
| **Transcript analysis** | ❌ | A transcription service (Whisper / `fal-ai` ASR / provider captions) + a `transcripts` table + an analysis prompt chain. No transcription anywhere today. |
| **Hook extraction** | ❌ | An LLM analysis pass over transcripts/captions to isolate first-3-second hooks; a `hooks` corpus. (Kimi can do the LLM part — the *data* and *pipeline* are missing.) |
| **Viral pattern analysis** | ❌ | Metrics-joined dataset (views/engagement velocity) + clustering/embedding + an analysis layer. Needs ingestion + **embeddings + a vector store** (none exist). |
| **Content knowledge base** | ❌ | A retrieval corpus: Postgres + **pgvector** (Supabase supports it) storing transcripts/hooks/patterns as embeddings; a retrieval API. Nothing today. |
| **Automated content generation** | 🟡 Partial | Generation exists (Kimi), but it's **topic-prompted, not knowledge-grounded**. Once a KB + retrieval exist, wire **RAG** so generation is conditioned on proven patterns. |

### What must be built (concrete component list)
1. **Ingestion services** — per-platform fetchers (YouTube Data API; Apify/3rd-party for TikTok/IG) writing to new tables `research_sources`, `research_items` (raw post + metrics).
2. **Transcription pipeline** — pull audio/captions → Whisper/ASR → `transcripts`.
3. **Vector layer** — enable **pgvector** on Supabase; add `content_embeddings`; choose an embeddings provider (OpenAI `text-embedding-3`, or fal/together). *(Note: Kimi/Moonshot is chat-only here; embeddings need a separate provider.)*
4. **Analysis chains** — LLM passes for hook extraction, structure tagging, and "why it worked" summaries → `hooks`, `patterns` tables.
5. **Knowledge base + retrieval API** — `/api/research/query` doing semantic search over embeddings.
6. **RAG-grounded generation** — extend `generate-script`/`generate-strategy` to retrieve top-K patterns/hooks for the niche and inject them into the prompt.
7. **Job queue + storage** — ingestion/transcription are long-running → Vercel Queues (or external worker) + Blob for raw media; cannot live in 30s request handlers.
8. **Research UI** — a new view to run/inspect research, browse the knowledge base, and "generate from pattern."

**Dependency order:** storage + queue → ingestion → transcription → embeddings/vector → analysis → KB/retrieval → RAG generation → UI. Items 1–3 are the foundation; nothing downstream is possible without them.
